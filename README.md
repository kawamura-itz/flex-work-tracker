# フレックス勤務トラッカー

フレックスタイム制向けの、自分用・ローカル保存の勤務時間管理ツール。
月の必要勤務時間を自動算出し、「あと何時間働けばよいか／あと何時間休めるか」を
過不足メーター（青＝余力・赤＝不足）で常時表示します。

- 完全クライアントサイド（SPA）。サーバーにデータを送りません。
- データはブラウザの IndexedDB に保存（端末ローカル）。
- GitHub Pages 等の静的ホスティングにそのまま載ります。

## 技術スタック

Vite + React + TypeScript / Dexie.js (IndexedDB) / date-fns / react-router (HashRouter)

## 開発

```bash
npm install
npm run dev        # 開発サーバ
npm run test       # ユニットテスト（計算ロジック）
npm run typecheck  # 型チェック
npm run build      # 本番ビルド → dist/
npm run preview    # ビルド結果をローカル確認
npm run holidays   # 内閣府CSVから祝日データを再生成（public/holidays.json）
```

## アーキテクチャ

| 層 | 場所 | 役割 |
|---|---|---|
| 型定義 | [src/types.ts](src/types.ts) | データモデル（§9）と既定設定 |
| ドメイン | [src/domain/](src/domain/) | 純粋関数：時間計算・休日判定・期間境界・過不足計算（§4, §8）|
| 永続化 | [src/db.ts](src/db.ts) | Dexie によるIndexedDB・JSONバックアップ |
| 状態 | [src/state/](src/state/), [src/hooks/](src/hooks/) | 設定・祝日・現在期間・タイマー |
| 画面 | [src/pages/](src/pages/), [src/components/](src/components/) | ホーム(過不足＋月カレンダー)/試算/履歴/設定 |

ホームは過不足メーターと **月カレンダー** が中心。カレンダーの日付をタップすると
その日の編集モーダル（[DayEditor](src/components/DayEditor.tsx)）が開き、区分や勤務時間を
入れると「この日の過不足への影響 ±」が即時に表示されます。開始/終了の初期値は設定の
**勤務開始時刻** と所定労働時間から決まります。各日は みなし / 記録 / 見込み / 休 で
色分けし、記録した日は労働時間（所定とズレた日だけ色付き）を表示します。

計算ロジックの中核は [src/domain/calc.ts](src/domain/calc.ts) にあり、
要件 §4.7 の検証例を含む31件のテスト（[src/test/](src/test/)）で固めています。

### 過不足の計算（要件 §4）

```
着地見込み = 実績(今日まで) + 計画(未来の記録) + 残り稼働日数 × 1日所定
過不足     = 着地見込み − 必要時間
```

実績は当日までの記録のみ、未来に入力済みの記録（計画有給など）は「計画」として
着地見込みに反映します（要件 §4.4 を一般化。未来の有給を記録しても着地見込みが
不自然に下がらないようにするため）。

### 所定時間みなし（おすすめ運用）

設定の「未記録の過去平日を所定時間とみなす」が **ON（既定）** のとき：

- 平日は **黙っていれば所定時間はたらいた扱い**（みなし）になります（昨日まで）。普通の日は入力不要。
- **有給・欠勤・早退（短縮）・残業など「何かあった日」だけ記録**すると、その差分だけ
  過不足がプラマイ動きます。所定どおりの日は記録してもしなくても過不足は変わりません。
- 今日・未来は「見込み」。**仮/確定の区別は設けず**、記録はいつでも編集できます
  （保存＝そのまま反映、削除すればみなしに戻る）。

この挙動が不要なら設定でOFFにできます（OFF＝明示的に記録した日だけ算入。
未記録の過去平日は「未確定（0扱い）」としてカレンダーに表示）。
計算ロジックは [src/domain/calc.ts](src/domain/calc.ts) の `computeStatus` / `classifyDay` /
`effectiveConfirmEnd`、テストは [src/test/calc.test.ts](src/test/calc.test.ts) を参照。

## GitHub Pages へのデプロイ

`vite.config.ts` の `base` は `'./'`（相対パス）なので、プロジェクトサイト
（`ユーザー名.github.io/リポジトリ名/`）でもカスタムドメイン（ルート配信）でも
そのまま動きます。クライアントルーティングは HashRouter を使っているため、
直リンクでの 404 も起きません。

同梱の [.github/workflows/deploy.yml](.github/workflows/deploy.yml) を使うと、
`main` への push でビルド＆Pages へ自動公開されます。
リポジトリの Settings → Pages → Source を「GitHub Actions」に設定してください。

## データの保存・移行

- データは IndexedDB（ドメイン単位）に保存されます。ドメインが変わると引き継がれません。
  移行の可能性があるならカスタムドメインの利用を推奨します。
- 初回に `navigator.storage.persist()` で永続化を要求します（設定画面から再要求可）。
- 設定画面から全データを **JSON でエクスポート／インポート** できます。
  端末移行・ブラウザのデータ削除に対する保険として定期的にエクスポートしてください。

## ライセンス / 注意

自分用ツール。GitHub Pages のパブリックリポジトリはコードが公開されますが、
利用者のデータはブラウザ内に留まり公開されません。商用利用（広告・課金）は
GitHub Pages の規約上不可です。
